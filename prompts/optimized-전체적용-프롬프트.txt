# VDOM 최적화 엔진 전체 적용 — 구현 프롬프트

## 배경

이 프로젝트는 바닐라 JS Virtual DOM 교육용 프로젝트이다.
현재 `src/optimized/` 폴더에 벤치마크 전용 최적화 모듈이 있으며,
이를 프로젝트 전체에 적용하는 작업이다.

핵심 최적화:
- key 기반 diff (index 순차 비교 → key 매칭)
- WeakMap O(1) patch (getNodeByPath O(n) 반복 → WeakMap 직접 조회)
- MOVE 패치 타입 추가 (순서 재배치 지원)

⚠️ 설계 원칙 (구현 전체에 적용):
nodeMap(WeakMap)은 현재 렌더 트리에 종속된 "캐시"다. 영속 상태 저장소가 아니다.
렌더 트리가 교체되는 시점(cloneVdom/undo/redo/reset/iframe snapshot)에서는
반드시 nodeMap을 폐기(new WeakMap())하고 vdomToDomMapped로 재구축해야 한다.
source of truth는 항상 currentVDOM + 실제 DOM이며, nodeMap은 조회 성능을 위한 캐시일 뿐이다.


## 절대 수정 금지 파일 (팀 계약)

## Phase 1: src/optimized/ 내부 수정 (기존 파일 무수정)
---

### 1-1. patch-mapped.js — 렌더 경로 통일 + nodeMap 갱신

문제: ADD/REPLACE에서 `vdomToDom`(기본)을 사용하여 새 DOM이 nodeMap에 등록 안 됨.
연속 패치 시 2회차부터 _ref 조회 실패.

수정:
- line 11: `import { vdomToDom } from "../vdom.js"` 삭제
- 대신: `import { vdomToDomMapped } from "./vdom-mapped.js"` 추가
- line 22 (ADD): `vdomToDom(patch.node)` → `vdomToDomMapped(patch.node, nodeMap)`
- line 39 (REPLACE): `vdomToDom(patch.node)` → `vdomToDomMapped(patch.node, nodeMap)`


### 1-2. diff-keyed.js — key 폴백 규칙

문제: getKey()가 props.key만 확인. domToVdom 결과에는 props["data-key"]로 들어옴.

수정 (line 31-33):
```js
// 기존
function getKey(vnode) {
  return vnode?.props?.key ?? null;
}
// 변경
function getKey(vnode) {
  return vnode?.props?.key ?? vnode?.props?.["data-key"] ?? null;
}
```


### 1-3. diff-keyed.js + patch-mapped.js — MOVE 패치 타입 추가

현재 key 매칭은 하지만 순서 재배치를 감지하지 못한다.

diff-keyed.js에 추가할 것:
- key 매칭 후 위치가 달라진 노드에 대해 MOVE 패치 생성
- LIS(Longest Increasing Subsequence) 알고리즘으로 최소 이동 계산
- 패치 포맷: `{ type: "MOVE", path, from, to, _ref, _parentRef }`

patch-mapped.js에 추가할 것:
- MOVE case: `parent.insertBefore(nodeMap.get(patch._ref), parent.childNodes[patch.to])`

주의: MOVE는 기존 5개 패치 타입(ADD/REMOVE/REPLACE/PROPS_UPDATE/TEXT_UPDATE)에
추가하는 것이므로 기존 동작에 영향 없어야 한다.

### 2-5. cloneVdom 후 nodeMap 재구축 전략

cloneVdom은 JSON.parse(JSON.stringify())를 사용하므로 WeakMap 키가 끊긴다.
main.js에 cloneVdom은 14곳 사용 중 (import 제외).

규칙: cloneVdom으로 새 vdom을 currentVDOM에 저장한 후,
다음 렌더 시 vdomToDomMapped가 호출되면 nodeMap이 자동 재구축된다.

따라서 cloneVdom 14곳 자체는 수정 불필요.
단, nodeMap을 초기화하는 시점을 명확히 해야 한다:

```js
// undo 시 (line 1160)
currentVDOM = cloneVdom(history[historyIndex]);
nodeMap = new WeakMap();  // ← 추가: 다음 렌더에서 재구축됨

// redo 시 (line 1180)
currentVDOM = cloneVdom(history[historyIndex]);
nodeMap = new WeakMap();  // ← 추가

// reset 1 (line 1193-1195)
history = [cloneVdom(initialVDOM)];
currentVDOM = cloneVdom(initialVDOM);
nodeMap = new WeakMap();  // ← 추가

// reset 2 (line 1224-1226)
history = [cloneVdom(initialVDOM)];
currentVDOM = cloneVdom(initialVDOM);
nodeMap = new WeakMap();  // ← 추가
```

핵심: nodeMap 초기화 후 iframe에 render-snapshot을 보내면,
iframe이 자체적으로 vdomToDomMapped를 실행하여 자기 nodeMap을 구축한다.

| 시나리오 | 파라미터 1 | 파라미터 2 |
|----------|-----------|-----------|
| S1 대량 수정 | 전체 항목 수 (기본 3000) | 변경 비율 % (기본 5) |
| S2 중간 삽입 | 전체 항목 수 (기본 1000) | 삽입 항목 수 (기본 100) |
| S3 속성 토글 | 전체 요소 수 (기본 1000) | — |
| S4 트리 리프 | 트리 깊이 (기본 6) | 분기 수 (기본 4) |

### 4-2. UI 구현 방식 — 숫자 입력 필드

슬라이더가 아닌 `<input type="number">` 사용. 이유:
- 정확한 값 입력 가능 (10000 같은 극단값 테스트)
- 팝업 공간 절약 (슬라이더 × 시나리오 수 = 너무 많음)
- 구현이 간단

### 4-3. 노드 폭발 방지 — 총 노드 수 제한

S4(트리 리프)는 breadth^depth 개의 노드를 생성한다.
사용자가 depth=10, breadth=6을 입력하면 6^10 ≈ 6천만 노드 → 브라우저 멈춤.

반드시 아래 안전장치를 구현해야 한다:

1) 총 노드 수 사전 계산 함수:
```js
function estimateNodeCount(depth, breadth) {
  // breadth^0 + breadth^1 + ... + breadth^depth = (breadth^(depth+1) - 1) / (breadth - 1)
  if (breadth <= 1) return depth + 1;
  return Math.floor((Math.pow(breadth, depth + 1) - 1) / (breadth - 1));
}
```

2) 노드 상한: 100,000개.
   파라미터 입력값이 변경될 때마다 예상 노드 수를 계산하여 표시.
   상한 초과 시:
   - 입력 필드에 경고 스타일 (빨간 테두리)
   - 실행 버튼 비활성화
   - "예상 노드 수: 1,234,567 (상한 100,000 초과)" 메시지 표시

3) S4 외 시나리오(S1, S2, S3)는 단순 리스트이므로 max 값(50,000)으로 충분.
   추가 계산 불필요.

4) S2 삽입 항목 수 검증: insertCount가 count보다 클 수 없음.
   입력 시 자동으로 min(insertCount, count) 적용.

구현 위치:
- 노드 수 계산: scenarios.js에 estimateNodeCount 함수 또는 각 시나리오에 estimate 콜백
- 검증 + 경고 UI: benchmark-popup.js의 파라미터 input 이벤트 핸들러
