[공통 규격 - 반드시 준수]
프로젝트 목표: 바닐라 JS로 Virtual DOM 핵심 흐름(VDOM 생성 -> diff -> patch -> DOM 반영 + undo/redo) 구현.

0) 협업 규칙
- 이 프로젝트는 4명이 병렬 작업 중입니다.
- 각 팀원은 자신의 담당 파일만 수정합니다. 다른 파일 수정 금지.
- 함수명, VDOM 스키마, patch 포맷은 임의 변경 금지.
- export / 전역 노출 방식은 현재 프로젝트 파일 구조를 그대로 따릅니다.

1) 파일 소유권
- 팀원 A: src/vdom.js
- 팀원 B: src/diff.js
- 팀원 C: src/patch.js
- 팀원 D: main.js, index.html, style.css, README.md

2) UI / 루트 계약
- 실제 영역(actual area)의 컨테이너 요소 자체는 VDOM에 포함하지 않습니다.
- 테스트 영역(test area)의 컨테이너 요소 자체도 VDOM에 포함하지 않습니다.
- VDOM은 항상 단일 루트 element 1개를 표현합니다.
- 사용자는 textarea에 HTML 문자열을 편집합니다.
- 테스트 영역은 textarea 입력을 파싱한 뒤, 그 결과로 만든 VDOM을 다시 렌더링한 미리보기 DOM입니다.
- top-level에서는 Element 1개만 허용합니다.
- top-level의 공백-only Text와 Comment는 무시합니다.
- top-level에 Element가 2개 이상 있거나, 공백-only가 아닌 Text가 있으면 잘못된 입력으로 간주하고 patch를 수행하지 않습니다.

3) VDOM 스키마
- 텍스트 노드: { type: "#text", text: string }
- 엘리먼트 노드: { type: string, props: Record<string,string>, children: VNode[] }

4) 의미 있는 노드 / 공백 텍스트 정책
- Element 노드는 포함합니다.
- Text 노드는 trim() === "" 이면 무시합니다.
- Comment 노드는 무시합니다.
- Text 노드를 유지할 때는 원본 text 값을 그대로 보존합니다. trim 결과로 저장하지 않습니다.
- path 인덱스는 raw childNodes 전체 기준이 아니라, 위 규칙을 통과한 의미 있는 자식 노드 순서 기준입니다.
- 따라서 domToVdom, diff, getNodeByPath, ADD 삽입 위치 해석은 모두 같은 기준을 사용해야 합니다.

5) Patch 포맷
- ADD: { type:"ADD", path:number[], node:VNode, index:number }
- REMOVE: { type:"REMOVE", path:number[] }
- REPLACE: { type:"REPLACE", path:number[], node:VNode }
- PROPS_UPDATE: { type:"PROPS_UPDATE", path:number[], setProps:Record<string,string>, removeProps:string[] }
- TEXT_UPDATE: { type:"TEXT_UPDATE", path:number[], text:string }

6) Patch path 규칙
- ADD의 path는 부모 노드 경로입니다.
- ADD의 index는 부모 아래의 의미 있는 자식 노드 기준 삽입 위치입니다.
- REMOVE / REPLACE / PROPS_UPDATE / TEXT_UPDATE의 path는 대상 노드 경로입니다.
- path [] 는 루트 노드를 의미합니다.
- 루트 element의 태그가 바뀌면 REPLACE at path [] 를 사용합니다.
- 이번 프로젝트에서는 최상위 루트 삭제(path [] 의 REMOVE)는 생성하지 않습니다. 루트는 항상 존재해야 합니다.

7) Patch 생성 / 적용 순서 규칙
- diff는 깊이 우선 순서로 patch를 생성합니다.
- 같은 부모 아래에서 여러 REMOVE가 필요하면, REMOVE patch는 반드시 더 큰 child index부터 생성합니다.
- 즉 같은 부모의 REMOVE는 index 내림차순이어야 합니다.
- ADD는 필요한 삽입 위치 기준으로 생성합니다.
- applyPatches는 전달받은 patch 배열의 순서를 그대로 적용하는 것을 기본으로 합니다.

8) DOM 정규화 / apply 계약
- applyPatches(rootDom, patches)는 patch 적용 후의 현재 실제 루트 DOM 노드를 반환합니다.
- rootDom 인자는 컨테이너가 아니라, 실제 영역 안에 렌더링된 현재 루트 노드입니다.
- rootDom은 첫 patch 전에 반드시 "정규화된 DOM" 이어야 합니다.
- 정규화된 DOM이란 domToVdom(rootDom) -> vdomToDom(vdom) 과 동일한 구조를 가진 DOM을 뜻합니다.
- 즉 무시 대상인 공백-only Text와 Comment가 제거된 canonical DOM만 patch 대상으로 사용합니다.
- path [] 에서 REPLACE가 일어나면 applyPatches는 새 루트 노드를 반환해야 합니다.
- 잘못된 path가 들어오면 앱이 죽지 않도록 방어적으로 처리합니다.
- 잘못된 path 처리 방식은 경고 로그 또는 조용한 무시 중 하나로 고정하고, 프로젝트 전체에서 일관되게 유지합니다.

9) 공통 함수명(이름 변경 금지)
- domToVdom(node)
- vdomToDom(vnode)
- diff(oldVDOM, newVDOM, path = [])
- applyPatches(rootDom, patches)
- getNodeByPath(root, path)
- cloneVdom(vdom)
- syncBothAreasFromVdom(vdom)

10) 상태 / history 규칙
- currentVDOM은 항상 실제 영역 루트 노드와 같은 상태를 나타내는 VDOM입니다.
- Patch 후 currentVDOM = cloneVdom(newVDOM)
- Undo / Redo 후에도 currentVDOM = cloneVdom(history[historyIndex]) 를 반드시 수행합니다.
- 새 상태 push 시 history는 historyIndex + 1 까지만 남기고 뒤쪽 redo 이력은 제거합니다.

11) textarea 동기화 규칙
- textarea는 사용자의 현재 편집 입력을 담습니다.
- init 시 textarea는 현재 실제 루트 DOM의 outerHTML로 초기화합니다.
- Undo / Redo 또는 syncBothAreasFromVdom 이후 textarea는 방금 렌더링된 실제 루트 DOM의 outerHTML로 다시 동기화합니다.
- Patch 버튼 성공 후에는 textarea 값을 강제로 다시 쓰지 않습니다. 사용자가 입력한 문자열을 그대로 유지합니다.

12) 범위 제한
- child index 기반 diff만 구현합니다. key / move 최적화는 제외합니다.
- Patch 버튼에서는 실제 영역에 대해 부분 업데이트를 수행합니다.
- 테스트 영역 미리보기와 Undo / Redo에서는 전체 재렌더를 허용합니다.
- MutationObserver는 필수 구현이 아닙니다.

13) 코드 스타일
- 순수 JS(ES6), 외부 라이브러리 금지
- 함수마다 짧은 JSDoc 작성
- 한계 사항은 주석으로 명시 가능

14) README / 발표 반영 포인트
- README에는 아래 내용을 반드시 포함합니다.
- 프로젝트 소개
- 실제 DOM이 느린 이유: reflow / repaint 관점
- Virtual DOM이 필요한 이유
- Diff 알고리즘의 5가지 핵심 케이스: 추가 / 삭제 / 교체 / 텍스트 변경 / 속성 변경
- Patch가 실제 DOM에 반영되는 방식
- Undo / Redo와 state history 구조
- 브라우저 API 관련 설명: Document / Window / DOMParser 또는 template
- 이번 구현은 Patch 버튼 기반 수동 반영임을 명시합니다.
- 테스트 케이스와 검증 결과를 표 또는 체크리스트 형태로 포함합니다.
