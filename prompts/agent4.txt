당신은 팀원 D입니다. UI 이벤트와 history 통합을 담당하세요.

[담당 파일]
- main.js, index.html, style.css, README.md 만 수정 가능
- src/vdom.js, src/diff.js, src/patch.js 는 수정하지 마세요

[담당 범위]
1) 상태 변수
- let history = []
- let historyIndex = -1
- let currentVDOM = null
- let actualRootNode = null

2) UI 구조
- 실제 영역 컨테이너 1개, 테스트 영역 컨테이너 1개, HTML 입력용 textarea 1개, Patch / Undo / Redo 버튼을 구성
- 필요하면 입력 오류 메시지를 보여줄 작은 안내 영역 1개를 추가 가능
- 사용자는 textarea에서 HTML 문자열을 수정합니다
- 테스트 영역은 textarea 입력을 파싱한 뒤, 그 결과 VDOM을 다시 렌더링한 미리보기 DOM입니다

3) 초기화(init)
- 실제 영역 컨테이너 안의 첫 번째 루트 element를 읽어 initialVDOM 생성
- initialVDOM = domToVdom(actualRootElement) 로 만든 뒤, 실제 영역 컨테이너를 비우고 vdomToDom(initialVDOM)으로 다시 렌더해 초기 실제 DOM을 1회 정규화합니다
- 즉, 무시 대상 공백-only Text와 Comment를 제거한 canonical DOM을 실제 patch 기준 루트로 만듭니다
- history = [cloneVdom(initialVDOM)], historyIndex = 0
- currentVDOM = cloneVdom(initialVDOM)
- actualRootNode는 정규화 후 실제 영역의 현재 루트 DOM 노드 참조를 저장
- 테스트 영역을 initialVDOM으로 전체 렌더
- textarea 값은 actualRootNode.outerHTML 로 초기화
- 버튼 활성 / 비활성 상태 갱신

4) textarea 파싱 / 검증
- textarea의 HTML 문자열을 DOMParser 또는 template 요소로 파싱
- top-level에서 공백-only Text와 Comment는 무시
- top-level Element가 정확히 1개이고, 그 외 의미 있는 top-level Text가 없을 때만 유효한 입력으로 판단
- 유효한 입력이면 parsedRoot -> domToVdom(parsedRoot) 로 newVDOM을 만든 뒤, 테스트 영역을 vdomToDom(newVDOM)으로 전체 렌더
- 테스트 영역도 항상 vdomToDom 결과를 사용해 canonical DOM으로 유지
- 유효하지 않으면 에러 메시지를 보여주고 Patch 버튼을 비활성화

5) Patch 버튼 핸들러
- 현재 textarea 입력을 다시 파싱 / 검증하여 newVDOM 생성
- 유효하지 않으면 patch를 수행하지 않음
- patches = diff(currentVDOM, newVDOM)
- actualRootNode = applyPatches(actualRootNode, patches)  // 실제 영역만 부분 업데이트
- currentVDOM = cloneVdom(newVDOM)
- history를 historyIndex + 1 까지 자른 뒤 push
- historyIndex를 마지막으로 이동
- 버튼 활성 / 비활성 상태 갱신

6) Undo / Redo 버튼
- Undo: historyIndex > 0 이면 -- 후 syncBothAreasFromVdom(history[historyIndex])
- Redo: historyIndex < history.length - 1 이면 ++ 후 syncBothAreasFromVdom(history[historyIndex])
- Undo / Redo 이후에는 반드시 currentVDOM = cloneVdom(history[historyIndex]) 수행

7) syncBothAreasFromVdom(vdom)
- 실제 영역 컨테이너와 테스트 영역 컨테이너를 모두 비우고 vdomToDom으로 전체 렌더
- 이때 실제 영역도 다시 canonical DOM으로 렌더되므로 이후 patch 대상 DOM은 계속 정규화된 상태를 유지합니다
- 렌더 후 actualRootNode를 새 실제 루트 노드로 다시 갱신
- textarea 값도 actualRootNode.outerHTML 로 다시 동기화
- 에러 메시지는 초기화하거나 현재 상태에 맞게 갱신
- 버튼 활성 / 비활성 상태도 함께 갱신

8) README / 문서화
- README에 아래 항목을 반드시 포함
  - 프로젝트 소개
  - 실제 DOM이 느린 이유(reflow / repaint)
  - Virtual DOM 구조와 필요한 이유
  - Diff 알고리즘의 5가지 핵심 케이스
  - Patch가 실제 DOM에 반영되는 방식
  - Undo / Redo와 state history 구조
  - 브라우저 API(Document, Window, DOMParser 또는 template)
  - 테스트 케이스 / 검증 결과
- README에는 textarea 기반 입력 방식을 선택한 이유도 짧게 설명
  - 이유: 텍스트 변경뿐 아니라 속성 변경, 태그 교체, 노드 추가 / 삭제를 안정적으로 재현하기 쉽기 때문
- README에는 실제 영역 DOM을 init 시 1회 정규화하는 이유도 짧게 설명
  - 이유: 공백-only Text / Comment를 제외한 path 규칙과 실제 patch 대상 DOM 구조를 일치시키기 위해서
- 최소 테스트 케이스 예시:
  - 텍스트 변경
  - 속성 변경
  - 노드 추가
  - 노드 삭제
  - 태그 교체
  - Undo / Redo
  - 공백 텍스트 노드 무시

[중요]
- 팀원 A / B / C 함수 호출 규약을 그대로 사용
- 함수명 / 데이터 포맷 임의 변경 금지
- Patch 버튼에서는 실제 영역에 대해 반드시 부분 업데이트가 보이게 유지
- applyPatches는 현재 실제 루트 DOM 노드를 반환한다는 공통 계약을 그대로 사용할 것
- 실제 영역의 patch 대상 DOM은 항상 정규화된 canonical DOM이어야 함을 유지할 것
- README에는 현재 구현 범위와 제한사항(key / move 미지원, MutationObserver 미적용 가능 등)도 명시
- MutationObserver는 필수 구현이 아니며, 적용하지 않으면 README에 Patch 버튼 기반 수동 반영이라고 분명히 적을 것
