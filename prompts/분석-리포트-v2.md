## 6. 추가 분석 3건

### 6-1. 벤치마크 공정성 문제 🔴

**현상**: 항목 100개 + 변경 100%에서도 VDOM이 항상 빠름

**원인**: DOM 측이 "전체 파괴 → 전체 재구축"이라는 최악의 시나리오를 측정

| 측정 대상 | DOM 측 | VDOM 측 |
|-----------|--------|---------|
| 100% 텍스트 변경 | `replaceChildren()` + 100개 `createElement` + `setAttribute` + `appendChild` | 100개 `TEXT_UPDATE` (= `node.textContent = "..."`) |
| 속성 100% 토글 | 전체 재생성 | 100개 `PROPS_UPDATE` (= `setAttribute` 호출) |

DOM이 느린 이유는 **element 생성 비용** (createElement + setAttribute + DOM 트리 삽입) vs **속성/텍스트만 변경** (textContent/setAttribute)의 차이.

**결론**: 벤치마크가 "일부러 느리게"는 아니지만, DOM 측이 "가장 비효율적인 방법"을 대표하는 것은 사실. 교육용으로는 적절하나, 공정한 비교를 위해 "DOM (타겟 업데이트)" 모드 추가를 권장.

자세한 분석은 아래 §6-1 확장 참조.

### 6-2. WeakMap 메모리 안전성 ✅

- WeakMap의 키(vnode 객체)가 참조 해제되면 엔트리 자동 GC
- `nodeMap = new WeakMap()` 리셋 시 이전 맵 전체가 GC 대상
- 임시 DOM 생성(lines 468, 862)의 클론 노드도 함수 종료 후 GC
- **메모리 누수 없음** 확인

### 6-3. 파일 구조 정리 상태

```
src/optimized/           ← 삭제 시 원복 용이 (각 파일에 제거 방법 주석)
├── index.js             ← 단일 진입점

```

## §6-1 확장: 벤치마크 공정성 심층 분석

### DOM 측이 하는 일 (lines 238-244)

```js
container.replaceChildren();              // ① 기존 DOM 전체 제거
const newDom = vdomToDom(modifiedVdom);    // ② 새 DOM 전체 생성
container.appendChild(newDom);            // ③ 새 DOM 삽입
void container.offsetHeight;              // ④ 강제 리플로우
```

이것은 "상태가 바뀌면 전부 다시 그린다"는 가장 원시적인 접근법.

### VDOM 측이 하는 일 (lines 290-300)

```js
const patches = diffKeyed(initialVdom, modifiedVdom);  // ① 변경점만 찾기
rootNode = applyPatchesMapped(rootNode, patches, nodeMap);  // ② 변경점만 적용
void container.offsetHeight;                           // ③ 강제 리플로우
```

### 왜 극단적 파라미터에서도 VDOM이 이기는가

**100개 항목 + 100% 변경 시:**

| 비용 항목 | DOM | VDOM |
|-----------|-----|------|
| 기존 노드 제거 | 100개 remove | 0 (제자리 수정) |
| 새 요소 생성 | 100× `createElement("li")` | 0 |
| 속성 설정 | 100× `setAttribute("key", ...)` | 0 |
| 트리 삽입 | 100× `appendChild` | 0 |
| 텍스트 변경 | 0 (새 노드에 포함) | 100× `textContent = "..."` |
| diff 비용 | 0 | O(n) 비교 |

핵심: `createElement` + `appendChild`는 브라우저 내부에서 **메모리 할당 + DOM 트리 구조 갱신 + 내부 인덱스 갱신**을 수반하지만, `textContent` 변경은 **기존 노드의 텍스트 데이터만 교체**하여 훨씬 가볍다.

### VDOM이 져야 하는 경우 (현재 미측정)

1. **완전 다른 트리 교체** — `<ul>` → `<table>` 같은 구조 변경 시 VDOM도 결국 REPLACE로 전체 재생성
2. **노드 수가 극소** (10개 미만) — diff 오버헤드 > DOM 재생성 비용
3. **초기 렌더링** — 둘 다 전체 생성이므로 VDOM은 diff 오버헤드만큼 불리

### 권장 개선안

"DOM (타겟 업데이트)" 모드 추가:
```js
// 예: 실제 DOM 개발자가 할 법한 최적화
for (let i = 0; i < count; i++) {
  if (shouldUpdate(i)) {
    container.children[i].textContent = newText;
  }
}
```

이렇게 하면 3자 비교가 가능:
- DOM (전체 재구축) — 가장 느림
- DOM (타겟 업데이트) — VDOM과 비슷하거나 더 빠를 수 있음
- VDOM (diff + patch) — 자동화된 최적 경로

이 비교가 "VDOM의 진짜 가치 = 개발자가 수동 최적화하지 않아도 자동으로 최적 경로를 찾아준다"를 더 잘 보여준다.
