# 🔍 Virtual DOM은 왜 빠른가?

> DOM이 결국 빨라야 하는 건 맞는데… 그렇다면 Virtual DOM은 대체 왜 필요한 걸까?

---

## 1. DOM이 느린 진짜 이유

많은 사람들이 "DOM 조작이 느리다"고 말하지만, 정확히 말하면 **DOM 조작 자체는 느리지 않다.**

```javascript
// 이 코드 자체는 매우 빠르다
element.textContent = "새로운 텍스트";
```

진짜 비싼 건 DOM을 건드린 **이후에** 브라우저가 수행하는 **렌더링 파이프라인**이다.

### 브라우저 렌더링 파이프라인

```
DOM 변경 → Style 계산 → Layout(Reflow) → Paint(Repaint) → Composite
```

| 단계 | 설명 | 비용 |
|------|------|------|
| **Style 계산** | 변경된 노드에 어떤 CSS가 적용되는지 다시 계산 | 중간 |
| **Layout (Reflow)** | 모든 요소의 위치와 크기를 다시 계산 | **매우 비쌈** |
| **Paint (Repaint)** | 계산된 레이아웃을 기반으로 픽셀을 다시 그림 | 비쌈 |
| **Composite** | 레이어를 합성하여 최종 화면 출력 | 상대적으로 저렴 |

**핵심 포인트:** DOM을 한 번 건드릴 때마다 이 파이프라인이 트리거될 수 있다. 특히 **Reflow**는 변경된 요소뿐 아니라 주변 요소의 레이아웃까지 연쇄적으로 재계산하기 때문에, 노드 수가 많을수록 비용이 기하급수적으로 증가한다.

```javascript
// Reflow를 강제로 발생시키는 패턴 (Layout Thrashing)
for (let i = 0; i < 1000; i++) {
  element.style.width = element.offsetWidth + 1 + "px";
  //                     ↑ 읽기: 강제 Reflow   ↑ 쓰기: Reflow 예약
  // 읽기와 쓰기가 번갈아 일어나면 매 루프마다 Reflow 발생!
}
```

## 2. 직접 DOM 조작의 문제점

### N개의 변경 = N번의 리플로우

직접 DOM을 조작하면, 변경사항 하나하나가 독립적으로 렌더링 파이프라인을 트리거한다.

```javascript
// ❌ 직접 DOM 조작: 매 변경마다 리플로우 가능성
function updateItems(items) {
  items.forEach((item, i) => {
    const el = document.getElementById(`item-${i}`);
    el.textContent = item.text;        // → Reflow 가능
    el.className = item.className;     // → Reflow 가능
    el.style.display = item.visible    // → Reflow 가능
      ? "block" : "none";
  });
}
```

3000개 항목을 업데이트하면? 최악의 경우 **3000번의 리플로우**가 발생할 수 있다.

### 배치 처리의 어려움

"그러면 `DocumentFragment`나 `requestAnimationFrame`으로 배치 처리하면 되지 않나?"

맞다. 가능하다. 하지만:

```javascript
// 수동 배치 처리 — 코드가 복잡해지고, 개발자가 직접 최적화를 관리해야 한다
function updateItemsManualBatch(items) {
  // 1. 변경 사항 수집
  const fragment = document.createDocumentFragment();
  const parent = document.getElementById("list");

  // 2. 기존 요소와 비교하여 뭘 바꿔야 하는지 직접 계산
  const oldChildren = Array.from(parent.children);
  items.forEach((item, i) => {
    if (oldChildren[i]?.textContent !== item.text) {
      // 3. 직접 비교 로직 작성...
      // 4. 직접 배치 적용 로직 작성...
    }
  });

  // 이 로직을 모든 UI 컴포넌트마다 반복해야 한다!
}
```

**직접 DOM 조작의 근본적 문제:**

- 🔴 "무엇이 변경되었는지"를 개발자가 직접 추적해야 한다
- 🔴 배치 처리를 매번 수동으로 구현해야 한다
- 🔴 컴포넌트가 복잡해질수록 최적화 코드가 비즈니스 로직을 압도한다
- 🔴 실수 하나가 전체 레이아웃 성능을 망칠 수 있다

## 3. Virtual DOM의 핵심 전략

Virtual DOM은 세 단계로 이 문제를 해결한다.

### Step 1: 메모리 상의 JS 객체를 먼저 수정

DOM 대신, 가벼운 JavaScript 객체를 먼저 조작한다.

```javascript
// Virtual DOM 노드 — 순수 JS 객체 (이 프로젝트의 VDOM 스키마)
const vnode = {
  type: "div",
  props: { class: "suspect-card", key: "suspect-1" },
  children: [
    { type: "h3", props: {}, children: [{ type: "#text", text: "김수상" }] },
    { type: "p",  props: {}, children: [{ type: "#text", text: "알리바이 불확실" }] }
  ]
};
```

이 객체를 아무리 수정해도 **브라우저 렌더링 파이프라인은 전혀 트리거되지 않는다.** JavaScript 엔진 내부의 메모리 연산일 뿐이다.

### Step 2: Diff 알고리즘으로 비교

이전 상태(Old VDOM)와 새 상태(New VDOM)를 비교하여, **정확히 무엇이 달라졌는지** 자동으로 계산한다.

```javascript
// 이 프로젝트의 diff 함수 (src/diff.js)
import { diff } from "./src/diff.js";

const patches = diff(oldVDOM, newVDOM);
// 결과: 패치 목록
// [
//   { type: "TEXT_UPDATE", path: [0, 1, 0], text: "알리바이 확인됨" },
//   { type: "PROPS_UPDATE", path: [0], setProps: { class: "cleared" }, removeProps: [] }
// ]
```

5가지(최적화 엔진은 6가지) 패치 타입으로 모든 변경을 표현한다:

| 패치 타입 | 설명 |
|-----------|------|
| `ADD` | 새 노드 추가 |
| `REMOVE` | 노드 삭제 |
| `REPLACE` | 노드 타입 자체가 변경 (예: `div` → `span`) |
| `PROPS_UPDATE` | 속성 변경 (class, style 등) |
| `TEXT_UPDATE` | 텍스트 내용 변경 |
| `MOVE` | 노드 위치 이동 *(최적화 엔진 전용)* |

### Step 3: 최소 변경만 Batch로 실제 DOM에 적용

계산된 패치를 **한 번에** 실제 DOM에 적용한다.

```javascript
// 이 프로젝트의 patch 함수 (src/patch.js)
import { applyPatches } from "./src/patch.js";

applyPatches(realDomRoot, patches);
// → 변경된 부분만 정확히 DOM에 반영
// → 리플로우는 최소 횟수만 발생
```

### 전체 흐름 요약

```
사용자 입력
    │
    ▼
┌─────────────────────────────┐
│  New VDOM 생성              │  ← 비용: 거의 0 (메모리 연산)
│  (JS 객체 조작)             │
└─────────────┬───────────────┘
              │
    ┌─────────┴─────────┐
    │                   │
    ▼                   ▼
┌─────────┐      ┌──────────┐
│ Old VDOM│      │ New VDOM │
└────┬────┘      └────┬─────┘
     │                │
     └───────┬────────┘
             │
             ▼
┌─────────────────────────────┐
│  Diff 알고리즘              │  ← 비용: 낮음 (JS 객체 비교)
│  "뭐가 달라졌지?"           │
└─────────────┬───────────────┘
              │
              ▼
       ┌─────────────┐
       │  Patches    │  ← 최소한의 변경 목록
       │  [n개]      │
       └──────┬──────┘
              │
              ▼
┌─────────────────────────────┐
│  Batch DOM 적용             │  ← 비용: 필요한 만큼만 (최소 Reflow)
│  실제 DOM에 한 번에 반영     │
└─────────────────────────────┘
```

**결과:** 3000개 요소 중 150개만 변경되었다면, 리플로우는 3000번이 아니라 **최소한의 횟수**만 발생한다.

## 4. 수사 보드 비유 🕵️

이 프로젝트는 **탐정 수사 보드**를 테마로 Virtual DOM을 설명한다. 이 비유가 핵심을 정확히 짚는다.

### 직접 DOM 조작 = 보드 전체를 매번 다시 만드는 수사관

> 용의자 한 명의 이름이 바뀌었다고 해서, 수사 보드에 붙은 **모든 사진, 모든 단서, 모든 연결선을 떼어내고 처음부터 다시 붙이는 것**과 같다.

```
단서 변경 발생!
    │
    ▼
┌──────────────────────────────────────────────────┐
│  수사 보드 전체 비우기                            │  😰 다 떼야 해...
│  ↓                                               │
│  용의자 A 사진 다시 붙이기                        │
│  용의자 A 이름 다시 쓰기                          │
│  용의자 B 사진 다시 붙이기   ← 안 바뀌었는데?     │
│  용의자 B 이름 다시 쓰기     ← 안 바뀌었는데?     │
│  단서 카드 30장 다시 붙이기  ← 안 바뀌었는데?     │
│  연결선 다시 그리기          ← 안 바뀌었는데?     │
└──────────────────────────────────────────────────┘
```

### Virtual DOM = 메모장에 먼저 정리하는 수사관

> 바뀔 내용을 **메모장(Virtual DOM)** 에 먼저 정리한다. 현재 보드와 메모장을 비교해서, **실제로 달라진 단서만** 보드에서 교체한다.

```
단서 변경 발생!
    │
    ▼
┌─────────────────────────────────────────────────────┐
│  📓 메모장에 새 상태 정리                            │  💡 일단 메모부터
│  ↓                                                  │
│  현재 보드 vs 메모장 비교 (Diff)                     │
│  ↓                                                  │
│  "용의자 A 이름만 바뀌었네!"                         │
│  ↓                                                  │
│  📌 보드에서 용의자 A 이름표만 교체                   │  ✅ 나머지는 그대로!
└─────────────────────────────────────────────────────┘
```

이 프로젝트의 UI도 이 비유를 그대로 구현한다:

- **왼쪽 패널 (현재 수사 보드)** = 실제 DOM이 렌더링된 화면
- **오른쪽 패널 (가설 보드)** = Virtual DOM으로 미리보기
- **패치 적용** = 바뀐 단서만 보드에서 교체하는 동작

수사 보드에 단서가 5개일 때는 전체를 다시 만들어도 괜찮다. 하지만 **단서가 3000개**라면? 바뀐 150개만 교체하는 것이 압도적으로 효율적이다.

## 5. 벤치마크로 증명 📊

이 프로젝트에는 Virtual DOM의 효율성을 직접 측정할 수 있는 벤치마크가 내장되어 있다 (`src/benchmark/scenarios.js`).

### 시나리오 1: 대량 수정 (Bulk Update)

**설정:** 3000개 항목 중 5%만 변경

```javascript
// 초기 상태: 3000개의 리스트 아이템
bulkUpdateInitial(3000)
// → <ul> 아래 3000개의 <li> 생성

// 변경 상태: 5%인 150개만 텍스트 수정
bulkUpdateModified(3000, 0.05)
// → 60번째마다 하나씩 "Updated-Item {i} ✓"로 변경
```

| 방식 | 동작 | 리플로우 영향 |
|------|------|--------------|
| **직접 DOM** | 3000개 전체를 새로 렌더링 | 🔴 3000개 노드 전체 Reflow |
| **Virtual DOM** | Diff로 150개 변경 감지 → 150개만 패치 | 🟢 150개 노드만 업데이트 |

**VDOM이 건드리는 DOM 노드 수: 전체의 5%**

### 시나리오 2: 중간 삽입 (Middle Insert)

**설정:** 기존 1000개 리스트의 중간에 500개 삽입

```javascript
// 직접 DOM: innerHTML로 1500개 전부 다시 생성
//   → 기존 1000개도 파괴 후 재생성
//   → 모든 이벤트 리스너, 스크롤 위치, 포커스 상태 소실

// Virtual DOM: diff 결과 500개의 ADD 패치만 생성
//   → 기존 1000개 요소는 그대로 유지
//   → 새 500개만 정확한 위치에 삽입
```

| 방식 | DOM 조작 횟수 | 기존 요소 |
|------|-------------|-----------|
| **직접 DOM** | 1500개 생성 | 🔴 전부 파괴 후 재생성 |
| **Virtual DOM** | 500개 삽입 | 🟢 기존 1000개 유지 |

### 시나리오 3: 부분 삭제 (Partial Delete)

**설정:** 3000개 중 30%인 900개 삭제

```javascript
// 직접 DOM: 전체를 다시 렌더링하거나, 삭제 대상을 직접 찾아야 함
// Virtual DOM: diff 결과 정확히 900개의 REMOVE 패치 생성

// 최적화 엔진(key-based diff)의 경우:
// → key를 기반으로 삭제 대상을 O(1)로 식별
// → WeakMap으로 해당 DOM 노드를 O(1)로 찾아 제거
```

| 방식 | 동작 | 남은 요소 |
|------|------|-----------|
| **직접 DOM** | 전체 재렌더링 또는 수동 탐색 | 🔴 2100개도 영향 받을 수 있음 |
| **Virtual DOM** | 900개 REMOVE 패치만 적용 | 🟢 2100개는 전혀 건드리지 않음 |

### 왜 차이가 나는가?

```
직접 DOM의 비용:
  전체 노드 수(N) × Reflow 비용

Virtual DOM의 비용:
  Diff 비용(N 비교) + 변경 노드 수(M) × Reflow 비용

N = 3000, M = 150 일 때:
  직접 DOM:  3000 × Reflow
  VDOM:     3000 × 비교(매우 저렴) + 150 × Reflow

  Reflow 비용 >> JS 객체 비교 비용 이므로,
  VDOM이 압도적으로 유리!
```

> **Diff 비용은 JavaScript 메모리 연산이라 매우 저렴하고, Reflow 비용은 브라우저 렌더링 엔진 작업이라 매우 비싸다.** 이 비용 차이가 Virtual DOM 전략의 핵심 근거다.

## 6. 결론

### DOM이 빨라야 하는 건 맞다

DOM API 자체의 속도를 탓하는 것이 아니다. `element.textContent = "x"` 한 줄은 충분히 빠르다.

### 문제는 "어떻게" 건드리느냐

같은 결과를 만들어도:

```javascript
// ❌ 3000번 건드리기
for (let i = 0; i < 3000; i++) {
  list.children[i].textContent = newData[i];  // 매번 Reflow 가능
}

// ✅ 바뀐 150개만 건드리기
patches.forEach(p => applyPatch(root, p));     // 최소 Reflow
```

**성능 차이는 수십 배에서 수백 배**까지 날 수 있다.

### Virtual DOM은 DOM을 대체하는 게 아니다

Virtual DOM은 DOM 위에 얹은 **전략 레이어**다.

```
┌─────────────────────────────┐
│  개발자의 의도               │  "이 데이터를 이렇게 보여줘"
└──────────────┬──────────────┘
               │
               ▼
┌─────────────────────────────┐
│  Virtual DOM 전략 레이어     │  "뭐가 바뀌었는지 계산하고,
│  (Diff → Patch → Batch)     │   최소한만 DOM에 반영할게"
└──────────────┬──────────────┘
               │
               ▼
┌─────────────────────────────┐
│  실제 DOM                   │  브라우저의 렌더링 엔진
│  (Reflow → Repaint)         │
└─────────────────────────────┘
```

수사관이 보드를 효율적으로 관리하듯, Virtual DOM은 **DOM을 효율적으로 사용하는 방법**이다.

> 🕵️ *수사 보드는 왜 바뀐 단서만 옮길까?*
>
> 전체를 다시 만들 시간에, 바뀐 것만 빠르게 교체하고 수사를 계속하는 게 낫기 때문이다.
> Virtual DOM도 같은 이유로 존재한다.
