# 벤치마크 공정성 분석 — 나이브 DOM vs VDOM

## 1. 비교 방식 요약

| 측 | 알고리즘 | 핵심 특성 |
|---|---|---|
| **DOM (나이브)** | `naiveDomUpdate()` — 위치(index) 기반 재귀 순회 | 기존 DOM 트리를 새 VDOM과 같은 위치끼리 비교, 다른 부분만 `textContent`/속성 수정 |
| **VDOM (최적화)** | `diffKeyed()` + `applyPatchesMapped()` | key 기반 diff → WeakMap O(1) 조회 → 패치 적용. MOVE/LIS 최적화 포함 |

> 이전 버전은 DOM 측이 "전체 파괴 + createElement 재구축"이라 항상 VDOM이 압도적으로 빨랐다.
> 현재는 DOM 측도 **부분 갱신**을 하므로 공정한 비교가 된다.

---

## 2. 시나리오별 결과와 원인

### S1: 대량 수정 — DOM이 빠름 ✅

```
3000개 리스트, 변경 비율 5% (150개 텍스트 변경)

DOM:  기존 노드를 순회하며 textContent가 다른 150개만 수정
VDOM: 3000개 전체를 diff 순회 → 150개 패치 목록 생성 → WeakMap 조회 → 수정
```

**원인**: 둘 다 결과적으로 같은 150개를 수정하지만,  
VDOM은 **diff 오버헤드**(전체 트리 순회 + 패치 객체 생성 + WeakMap 조회)가 추가된다.  
위치가 바뀌지 않는 시나리오에서는 위치 기반 비교가 최단 경로.

---

### S2: 중간 삽입 — 조건부 (변경 수에 따라 다름)

```
1000개 리스트 중간(500번째)에 100개 삽입

나이브 DOM (위치 기반):
  - index 0~499: 동일 → 스킵
  - index 500~599: 기존 row-500~599 vs 새 new-0~99 → textContent 수정 (100개)
  - index 600~999: 기존 row-600~999 vs 새 row-500~899 → textContent 수정 (400개)  
  - index 1000~1099: 기존 노드 없음 → appendChild (100개)
  총: 500개 textContent 수정 + 100개 생성 = 600개 조작

VDOM (key 기반):
  - key row-0~499: 동일 위치 → 스킵
  - key new-0~99: 새 노드 → ADD (100개)
  - key row-500~999: key로 식별 → 위치만 밀림, 내용 동일 → 안 건드림
  총: 100개 ADD = 100개 조작
```

**삽입 수가 적을수록** → DOM은 여전히 "밀린 뒤쪽 전부 수정"하지만 VDOM은 삽입분만 → **VDOM 유리**  
**삽입 수가 많을수록** → 둘 다 많은 작업 + VDOM diff 오버헤드 → **DOM과 비슷하거나 DOM 유리**

---

### S3: 속성 토글 — DOM이 빠름 ✅

```
1000개 요소의 class/data-active 일괄 변경

DOM:  순회하며 getAttribute 비교 → setAttribute 1000회
VDOM: 1000개 diff → PROPS_UPDATE 패치 1000개 생성 → WeakMap 조회 → setAttribute 1000회
```

**원인**: 100% 변경이라 VDOM의 "안 바뀐 걸 건너뛰는" 장점이 0.  
diff가 순수 오버헤드가 된다.

---

### S4: 트리 리프 — DOM이 빠름 ✅

```
깊이 6, 분기 4 트리의 모든 리프 텍스트 변경

DOM:  재귀 순회하며 리프의 textContent 비교·수정
VDOM: 전체 트리 재귀 diff → TEXT_UPDATE 패치 생성 → 수정
```

**원인**: S1과 동일 원리. 트리 구조가 변하지 않고 값만 바뀌면,  
직접 순회가 diff 단계를 건너뛰는 만큼 빠르다.

## 3. 핵심 결론

```
┌─────────────────────────────────────────────────────────┐
│  VDOM이 빠른 조건:                                       │
│  ✅ 노드의 순서가 바뀔 때 (삽입, 삭제, 이동)               │
│  ✅ 변경 비율이 낮고 전체 노드가 많을 때                    │
│  ✅ key로 동일 노드를 식별해서 건너뛸 수 있을 때             │
│                                                         │
│  DOM(나이브)이 빠른 조건:                                 │
│  ✅ 같은 위치에서 값만 바뀔 때 (순서 불변)                  │
│  ✅ 변경 비율이 높을 때 (100%면 diff가 순수 오버헤드)       │
│  ✅ 트리 구조가 고정이고 리프만 갱신할 때                   │
└─────────────────────────────────────────────────────────┘
```

## 4. VDOM의 진짜 가치

성능만 보면 VDOM이 항상 이기는 것은 아니다.  
**VDOM의 핵심 가치는 성능이 아니라 생산성:**

1. **선언적 UI** — "이 상태일 때 UI는 이것" 이라고만 작성하면 됨
2. **변경 추적 자동화** — 개발자가 "어떤 노드를 어떻게 바꿔라"를 수동 관리할 필요 없음
3. **안전한 일괄 갱신** — 복잡한 상태 변화에서도 누락/중복 갱신 없음

> React가 VDOM을 쓰는 이유도 "빠르니까"가 아니라 "편하니까"이다.  
> 성능이 목적이면 수동 DOM 조작이 항상 최적이지만, 유지보수와 확장성은 포기해야 한다.

## 5. innerHTML 토글이 빠졌는데?

이전 버전에서는 DOM 측에 "우리 DOM API vs 브라우저 innerHTML" 토글이 있었다.

**제거한 이유:**
- `innerHTML`도 `createElement`도 결국 "전체 재구축"이라 공정하지 않았음
- 둘 다 VDOM의 "부분 갱신"과는 다른 카테고리의 작업
- 현재의 `naiveDomUpdate()`가 VDOM과 **동일한 "부분 갱신" 카테고리**에서 비교하므로 더 공정

## 6. 노드 상한 (100,000개)

| 노드 수 | 예상 동작 |
|---|---|
| ~50,000 | 정상 작동, 벤치마크 1~3초 |
| 50,000~100,000 | 느려짐 체감, 5초+ 가능 |
| 100,000 이상 (차단됨) | 메모리 500MB+, GC 정지, 탭 크래시 위험 |

상한 100,000개는 "불쾌하지만 크래시는 안 나는" 경계선이다.
